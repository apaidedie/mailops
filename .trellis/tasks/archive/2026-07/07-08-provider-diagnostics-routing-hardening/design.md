# Provider Diagnostics and Routing Hardening Design

## Architecture

Provider routing remains owned by `outlook_web.services.provider_catalog`. Controllers and frontend modules consume projections from the catalog and must not rebuild provider capability or selection rules.

The new contract is `readiness_summary.routing_matrix`. It is a versioned, secret-free projection generated by `get_mailbox_provider_readiness_summary()` from the same `selection_policy`, `provider_integration_guide`, `provider_diagnostics`, and endpoint map already used by the provider discovery payloads.

## Data Contract

`routing_matrix.version` is `1`.

`routing_matrix.scopes` contains one entry for each supported provider-selection scope: `temp_runtime_default`, `task_temp_apply`, `pool_claim_default`, and `explicit_pool_claim`.

Each scope entry contains `scope`, `label`, `request_field`, `settings_key`, `env`, `endpoint`, `allowed_values`, `counts`, and `providers` where applicable. Provider rows contain `provider`, `label`, `kind`, `active`, `configured`, `usable`, `status`, `reason`, `aliases`, and `endpoints`.

The matrix can include alias rows such as `gptmail` or `imap` when they appear in selection-policy allowed values. Alias rows resolve display metadata from deployment aliases and guide provider entries while staying secret-free.

## Data Flow

Provider catalog and deployment profile build the allowed values and aliases. The provider integration guide builds per-provider readiness, deployment, and endpoint metadata. The readiness summary joins those sources into provider rows and the new routing matrix. `/api/providers`, `/api/external/providers`, `/api/external/capabilities`, `/api/mailboxes`, `/api/external/mailboxes`, and OpenAPI continue to receive this projection from shared helpers.

The unified mailbox UI reads `readinessSummary.routing_matrix` and renders compact selector metadata inside the provider readiness band. It does not decide provider usability by provider name.

## Compatibility

The change only adds fields. Existing readiness fields remain stable. Secret values remain excluded; only key names and public endpoint paths may appear.

## Rollback

If the matrix causes a regression, remove the new projection from `get_mailbox_provider_readiness_summary()`, remove the OpenAPI schema additions, and remove UI matrix rendering. Existing readiness summary consumers keep working because the old fields are unchanged.
