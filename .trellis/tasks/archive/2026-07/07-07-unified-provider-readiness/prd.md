# Unified mailbox provider readiness

## Goal

Expose a concise, secret-free provider readiness summary inside the unified mailbox directory so operators and external integrators can see inventory, local configuration, provider routing, and external entry points from one payload.

## Confirmed Facts

- `mailops.services.mailbox_catalog.list_unified_mailboxes()` already returns one directory shape for Outlook/IMAP account mailboxes and user-visible temp mailboxes.
- `provider_context` is generated by `mailops.services.provider_catalog.get_mailbox_directory_provider_context()` and is already shared by admin `/api/mailboxes` and external `/api/external/mailboxes` through the directory service.
- `provider_context.provider_diagnostics` intentionally exposes only summary, filter, defaults, and scope in mailbox directory payloads; it must not include the full per-provider diagnostics list there.
- `provider_context.provider_integration_guide.providers` already contains per-provider active/configured/readiness state, operation capabilities, request fields, and endpoint hints.
- The unified mailbox UI already renders provider policy and provider capability matrix from `provider_context`, but it does not expose a direct inventory-aware readiness summary that combines mailbox counts with provider readiness.
- OpenAPI currently types `MailboxProviderContext` as a loose object for provider diagnostics and does not require a readiness summary field.

## Requirements

- Add `provider_context.readiness_summary` as the stable mailbox-directory readiness projection.
- Build the summary from the same provider diagnostics and integration guide objects already used by `provider_context`; do not rebuild provider rules in controllers or frontend code.
- Include aggregate inventory counts by provider using directory mailbox facets or loaded mailbox items, and readiness/configuration counts by provider from provider discovery metadata.
- Keep the summary secret-free. It may expose provider keys, labels, endpoint paths, env/settings key names, and counts, but no API keys, bearer token values, passwords, JWTs, refresh tokens, task tokens, or consumer keys.
- Expose the field through both admin `/api/mailboxes` and external `/api/external/mailboxes` because both endpoints share the directory service.
- Update OpenAPI `MailboxProviderContext` so generated clients know `readiness_summary` is present.
- Update the unified mailbox UI to display the new readiness summary without adding provider-specific branches or a second provider registry.
- Do not change provider selection priority, env key names, request field names, database schema, provider health network probing, or mailbox message-read behavior.

## Acceptance Criteria

- `provider_context.readiness_summary.version == 1` appears in `/api/mailboxes` and `/api/external/mailboxes` responses.
- The summary includes `overall_status`, aggregate totals, issue counts, source priority, external endpoints, provider selector fields, and per-provider rows with mailbox inventory counts and readiness status.
- Account-backed and temp-mail providers both appear when present in the directory or provider guide.
- The directory payload remains secret-free in tests even when temp-mail records contain task tokens, JWT-like values, provider secrets, or consumer keys in stored metadata.
- OpenAPI marks `readiness_summary` as required under `MailboxProviderContext` and provides a typed schema for the summary.
- Frontend contract tests prove the unified mailbox UI consumes `providerContext.readiness_summary`, renders readiness summary/status/rows, and does not branch on built-in provider names or read settings credential inputs in that renderer.
- Focused backend/API/frontend contract tests pass, plus `node --check` for changed JavaScript and `git diff --check`.
