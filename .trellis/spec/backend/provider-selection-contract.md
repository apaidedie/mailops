### 3. Contracts

Provider discovery endpoints must expose top-level `selection_policy`, `provider_integration_guide`, `integration_manifest`, `quickstart`, and `documentation`. The guide must be generated by `get_provider_integration_guide()` from the same catalog, deployment profile, selection policy, active allowlist, diagnostics, and endpoint map as the owning payload. The manifest must be generated by `get_external_integration_manifest()` from that same guide and policy context, not rebuilt in controllers. `quickstart` must be generated inside that manifest and copied from `integration_manifest.quickstart` by `/api/external/capabilities`, `/api/external/providers`, and authenticated `/api/providers`; controllers and frontend code must not rebuild it. The documentation object must be generated by `get_provider_documentation_contract()` and reused by the top-level discovery payload, the guide, the manifest, and mailbox `provider_context` so local doc paths cannot drift between endpoints.

Authenticated admin `GET /api/providers` must also expose operator-facing default temp provider fields aligned with external providers discovery: `default_temp_mail_provider`, `default_temp_mail_provider_label`, `default_temp_mail_provider_configured`, `default_temp_mail_provider_missing_config`, and `default_temp_mail_provider_env` (`TEMP_MAIL_PROVIDER`). The default key must come from `get_operator_temp_mail_default_provider()` so bridge dual-register stored names project to the same collapsed `legacy_bridge` key present in `provider_integration_guide.providers`.

External API discovery uses versioned v1 paths only. `get_external_api_endpoint_map()` returns `/api/v1/external/*` paths and every discovery-owned endpoint map, provider guide, integration manifest, quickstart, mailbox directory provider context, readiness routing matrix, action contract, and mailbox session next-action payload must use those canonical paths. `get_external_api_legacy_endpoint_map()` returns `{}`. Capabilities and provider discovery must report `compatibility.legacy_supported=false`, empty `legacy_endpoints` / `aliases`, `compatibility.canonical_prefix=/api/v1/external`, `compatibility.legacy_prefix=/api/external` (removed prefix name for migration), and `compatibility.migration_doc=docs/migration/remove-legacy-external-api.md`.

All external API route handlers must be mounted only on `/api/v1/external/*` via `add_external_api_url_rule`. Do not re-introduce dual mounts for `/api/external/*` without an explicit product decision. Legacy `/api/external/*` must 404.

`GET /api/external/docs` and `GET /api/v1/external/docs` must expose the same authenticated first-party HTML documentation page. The page must use the same API-key guard and external guard chain as OpenAPI, render from the current OpenAPI contract and capabilities metadata, require no external CDN or package, and remain secret-free: it may show placeholder auth such as `X-API-Key: <your-api-key>` and endpoint paths, but must not echo real API keys, provider bearer tokens, passwords, refresh tokens, task tokens, consumer keys, or provider secret values.

`GET /api/external/openapi.json` and `GET /api/v1/external/openapi.json` must expose the same OpenAPI document. OpenAPI `paths` must contain canonical `/api/v1/external/*` operations only, including `/api/v1/external/docs`; do not duplicate every legacy operation body in `paths`. Legacy compatibility must be represented with machine-readable extensions such as `x-legacy-endpoints` and `x-compatibility`, copied from the capabilities compatibility contract. The document must expose the external endpoint map through `x-capabilities.selection_policy`, `x-capabilities.integration_manifest`, and `x-capabilities.documentation` and must include an `ApiKeyAuth` security scheme using the `X-API-Key` header.

`GET /api/external/health` must keep the existing `HealthData` fields and add a required `readiness` object generated by `get_external_api_readiness_summary()`. Health readiness is a compact, secret-free projection of the same local discovery state used by capabilities and providers; it must not rebuild provider rules in `controllers/system.py`. `readiness.status` is `ready` or `degraded`. `readiness.discovery.next_endpoints` must point callers to capabilities, providers, mailbox directory, and OpenAPI endpoints. `readiness.providers` must expose only aggregate readiness counts, active allowlist mode, and active allowlist values. `readiness.mailbox_directory` must expose the `/api/external/mailboxes` endpoint, `ready`, `empty`, `restricted`, or `degraded` status, an account-scope flag, compact summary counts, totals for all/account/temp mailboxes, and one-item quick-probe params. It must reuse the unified mailbox directory inventory path after applying the current external API consumer `allowed_emails` scope and must not include mailbox item rows. `readiness.pool` must report `ready`, `disabled`, `restricted`, `degraded`, or `unknown` based on external-pool enablement, current API key `pool_access`, and available pool lifecycle features. `readiness.task_temp_mailbox` must report the task-temp apply/finish endpoints and the `provider_name` selector field. It must never include full provider diagnostic entries, mailbox credentials, task tokens, consumer keys, JWTs, refresh tokens, provider bearer tokens, or other secret values.

`GET /api/external/health` may reuse the existing cached instance upstream probe summary, but it must not call provider upstream networks. Explicit provider network probing remains owned by `/api/external/providers/{kind}/{provider}/health?probe_network=true`.

`GET /api/providers/preflight` and `GET /api/external/providers/preflight` must expose the batch provider readiness preflight generated by `get_mailbox_provider_preflight()`. The admin route returns `{success: true, provider_preflight: ...}` and requires login. The external route returns the same object as `data` under the existing API-key envelope. Default preflight is local-only: it composes provider diagnostics, default diagnostics, selection policy, integration guide, readiness summary, and single-provider health rows without instantiating temp-mail providers or calling upstream networks. `probe_network=true` is explicit and may delegate only locally ready temp providers to `get_mailbox_provider_health(..., probe_network=True)`; account providers remain local-only because provider-wide Graph/IMAP probes are not meaningful without a concrete mailbox account. The preflight object must include `version`, `status`, `scope`, `summary`, `issues`, `defaults`, `filter`, `endpoints`, `providers`, `readiness_summary`, and `documentation`. It may expose secret key names but must never expose provider API key values, bearer token values, passwords, refresh tokens, provider JWTs, task tokens, consumer keys, or provider secret values.

`GET /api/external/openapi.json` must document `HealthData.properties.readiness` as `ExternalReadinessSummary`, and that schema must require `status`, `database`, `upstream_probe`, `discovery`, `providers`, `mailbox_directory`, `pool`, `task_temp_mailbox`, and `warnings`. `ExternalReadinessSummary.properties.mailbox_directory` must point to a typed `ExternalReadinessMailboxDirectory` schema that requires `status`, `endpoint`, `scoped`, `summary`, `totals`, and `quick_probe_params`.

`GET /api/external/openapi.json` must expose `/api/external/capabilities` with a typed `CapabilitiesData` schema instead of a loose object. That schema must require `service`, `version`, `public_mode`, `features`, `available_features`, `restricted_features`, `defaults`, `deployment_env`, `deployment_profile`, `selection_policy`, `provider_integration_guide`, `integration_manifest`, `quickstart`, `documentation`, `provider_filter`, `provider_diagnostics`, `endpoints`, `mailbox_directory`, `external_mailbox_read_contract`, `mailbox_session`, `pool`, and `task_temp_mailbox`, and must point discovery sub-objects to `ExternalCapabilitiesDefaults`, `ExternalEndpointMap`, `ProviderSelectionPolicy`, `ProviderIntegrationGuide`, `IntegrationManifest`, `IntegrationQuickstart`, `ProviderDocumentation`, `MailboxDirectoryDiscovery`, `MailboxSessionDiscovery`, `PoolDiscovery`, and `TaskTempMailboxDiscovery`.

`POST /api/external/mailbox-sessions/start` is the provider-neutral mailbox session entry point for external callers. It must be protected by the existing API-key auth and external guard chain. The request body must be a JSON object with required `caller_id` and `task_id`, plus optional `source_strategy`, `provider`, `provider_name`, `email_domain`, `project_key`, `prefix`, and `domain`. `source_strategy` defaults to `pool_first` and must be one of `pool_first`, `task_temp_first`, `pool_only`, or `task_temp_only`. `provider` selects pool claim providers; `provider_name` selects task temp-mail providers. If `provider_name` is omitted, task temp-mail creation may use its normal runtime default.

The session start response `data` must be a `MailboxSessionData` object with `session_type`, `email`, `provider`, `provider_label`, `read_capability`, `created_at`, `lifecycle`, `external_mailbox_read_contract`, and `next_actions`. Pool sessions use `session_type=pool_claim` and return existing pool lifecycle handles including `account_id`, `claim_token`, `claimed_at`, `lease_expires_at`, `complete_endpoint`, and `release_endpoint`. Task temp-mail sessions use `session_type=task_temp_mailbox` and return existing task lifecycle handles including `task_token`, `finish_endpoint`, `visible_in_ui`, and `status`. These lifecycle handles are intentionally caller-held action tokens, but the response must never include mailbox passwords, refresh tokens, provider bearer token values, provider JWTs, API keys, consumer keys, or provider secret values.

Session start must orchestrate existing services instead of creating a second mailbox lifecycle model. Pool sessions call the existing pool claim service. Task temp-mail sessions call `TempMailService.apply_task_mailbox`. `pool_first` may fall back to task temp-mail only when external pool is disabled or the pool claim service returns no available account. It must not fall back after validation, authorization, provider configuration, upstream, public-mode feature-disabled, or internal errors. `pool_only` never falls back. `task_temp_only` never calls pool. `task_temp_first` is reserved as a pool-including strategy in the public contract, so multi-key callers without pool access must not use it even while the first implementation starts with task temp-mail only.

`POST /api/external/mailbox-sessions/read` is the provider-neutral read endpoint for sessions created by `POST /api/external/mailbox-sessions/start`. It must be protected by API-key auth and external API guards. The request body must be a JSON object with required `session_type`, `read_action`, `caller_id`, and `task_id`. `session_type` is `pool_claim` or `task_temp_mailbox`. `read_action` is one of `messages`, `latest_message`, `message_detail`, `message_raw`, `verification_code`, `verification_link`, or `wait_message`. Pool session reads require `claim_token` and must reuse the existing claim-token read scope so claimed-at baseline filtering and claim read logging stay consistent. Task temp-mailbox session reads require `task_token`, must verify that the mailbox `consumer_key` matches the current API key consumer, and then read by the resolved email through the existing mailbox resolver. The endpoint may accept common read filters (`email`, `message_id`, `folder`, `skip`, `top`, `from_contains`, `subject_contains`, `since_minutes`, `code_length`, `code_regex`, `code_source`, `timeout_seconds`, `poll_interval`, and `mode`) but must not create a persisted session table, bypass existing mailbox resolver authorization, or introduce provider-specific read branches.

The session read response `data` must be a secret-free `MailboxSessionReadData` object with `session_type`, `read_action`, `email`, `status`, and `result`. `result` preserves the underlying read action shape: message list, message summary, message detail, raw message projection, verification result, or async probe result. The response must not echo claim tokens, task tokens, consumer keys, API keys, mailbox passwords, refresh tokens, provider bearer token values, provider JWTs, or provider secret values. Public-mode feature disables for raw content and wait-message must apply to this endpoint just as they apply to the direct external read endpoints.

`POST /api/external/mailbox-sessions/close` is the provider-neutral lifecycle close endpoint for sessions created by `POST /api/external/mailbox-sessions/start`. It must be protected by API-key auth and external API guards. The request body must be a JSON object with `session_type` equal to `pool_claim` or `task_temp_mailbox`, plus `caller_id` and `task_id`. Pool closes accept `account_id`, `claim_token`, `result`, optional `detail`, and optional `reason`. `result=release` routes to the existing `release_claim()` service; any other pool result routes to the existing `complete_claim()` service and must use the same runtime `VALID_RESULTS` validation. Task temp-mail closes accept `task_token`, optional `result`, and optional `detail`, verify the task mailbox `consumer_key` matches the current API key consumer, then call `TempMailService.finish_task_mailbox()` and cancel pending probes for that mailbox. The close endpoint must not create a persisted session table, mutate pool/temp rows directly, or bypass existing pool ownership, task ownership, public-mode feature disable, or external pool enablement rules.

The session close response `data` must be a secret-free `MailboxSessionCloseData` object with `session_type`, `close_action`, `status=closed`, and source-specific identifiers/status such as `account_id`, `pool_status`, `task_token`, and `email`. These lifecycle handles are caller-held action handles, but the response must never include mailbox passwords, refresh tokens, provider bearer token values, provider JWTs, API keys, consumer keys, or provider secret values.

`documentation.version` is `1`. It is a secret-free pointer contract for humans and generated clients. It must include stable entries for provider onboarding, external integration quickstart, plugin extension, plugin prompt, `.env.example`, provider config JSON, provider config TOML, the OpenAPI endpoint, and the authenticated first-party API docs page. Documentation entries may expose repository-relative paths, endpoint paths, legacy endpoint paths, labels, types, and purpose text; they must not expose API keys, bearer token values, provider credentials, passwords, JWTs, refresh tokens, task tokens, or consumer keys.

`provider_integration_guide.version` is `1`. It exposes source priority, secret policy, workflow instructions, alias maps, provider filter context, endpoint map, documentation, and per-provider entries. Per-provider entries must include provider key, label, kind, active/configured/readiness state, required/optional env keys, required/settings keys, secret key names, activation/default/request examples, mailbox directory filter, health endpoint, aliases, operation capabilities, and `contract_validation`. It may expose secret key names such as `DUCKMAIL_BEARER_TOKEN`; it must not expose secret values, bearer token values, API key values, passwords, JWTs, task tokens, or consumer keys.

Compatible Temp Mail Bridge dual-register keys (`custom_domain_temp_mail` and `legacy_bridge`) remain in the full `mailbox_providers` catalog and registry for stored-source compatibility. Operator-facing discovery projections — `provider_diagnostics.providers` and `provider_integration_guide.providers` — must collapse those keys to a single `legacy_bridge` row so external clients do not see two identical-label providers. Pool-claim alias maps may still list both source names. Diagnostics `summary.total` / `summary.temp` must match the collapsed provider list.

Operator/API discovery defaults (`defaults.temp_mail_provider`, `default_temp_mail_provider`, and `provider_diagnostics.defaults.temp_mail_provider.provider`) must project the same bridge canonical key via `get_operator_temp_mail_default_provider()`. Stored/runtime settings may still use `custom_domain_temp_mail`; diagnostics may expose that historical value as `raw_provider` while `provider` / `canonical_provider` are `legacy_bridge`. The projected default must exist in the collapsed guide provider list.

Active allowlist matching for Compatible Temp Mail Bridge must treat dual-register catalog keys (`custom_domain_temp_mail`, `legacy_bridge`) and runtime aliases (`gptmail`, `legacy_gptmail`, `temp_mail`) as one family: allowlisting any member keeps both catalog keys active via `_active_matches_gptmail_family` / `is_mailbox_provider_active`. Empty allowlist still means all providers active.

`integration_manifest.version` is `1`. It is the machine-readable starter contract for external projects and must include `auth`, `discovery`, `quickstart`, `selection`, `deployment`, `secret_policy`, `workflows`, `providers`, and `documentation`. `auth.header` is `X-API-Key`, `auth.placeholder` is `<your-api-key>`, and real API key values must never be echoed. `discovery.recommended_sequence` must include capabilities, providers, and mailbox directory discovery; `discovery.endpoints` must include the external endpoint map available to the owning payload, including `docs=/api/v1/external/docs` and `provider_preflight=/api/v1/external/providers/preflight` for canonical payloads. `quickstart.version` is `1` and is the compact first-step contract for generated clients. It must include placeholder API key auth, recommended discovery order, stable endpoint paths, provider selector fields, minimal request examples, and workflow keys including `start_mailbox_session` and `close_mailbox_session`. Its provider selector fields and allowed values must be derived from `selection_policy.scopes.explicit_pool_claim` and `selection_policy.scopes.task_temp_apply`; its request examples must not include provider secret key names or secret values. `selection.source_priority` must equal `selection_policy.source_priority`, and provider override request fields must come from `selection_policy.scopes.explicit_pool_claim.request_field` and `selection_policy.scopes.task_temp_apply.request_field`. `workflows` must be generated from the same endpoint map, selection policy scopes, and `get_external_mailbox_read_contract()` lifecycle contracts as the owning payload; when a custom endpoint map is passed, every workflow step must use that map consistently instead of mixing default canonical paths with caller-provided paths. It must cover mailbox session start/read/close, discovery, unified mailbox directory browsing, pool claim read/complete/release, and task temp-mail apply/read/finish with stable workflow keys and ordered step objects. Workflow provider selector fields and allowed values must be copied from `selection_policy.scopes.explicit_pool_claim` and `selection_policy.scopes.task_temp_apply`; do not maintain a second provider enum. Per-provider manifest entries are projections of `provider_integration_guide.providers`: secret env/settings keys are allowed as key names, but their hint `value` must be an empty string; non-secret defaults such as `MAILTM_API_BASE=https://api.mail.tm` and `DUCKMAIL_API_BASE=https://api.duckmail.sbs` may appear as defaults, and `contract_validation` must be copied from the guide entry. Do not add provider-name conditionals to build the manifest.

`GET /api/external/openapi.json` must type provider-selecting request bodies instead of using loose objects. `PoolClaimRequest.properties.provider.enum` and `MailboxSessionStartRequest.properties.provider.enum` must be derived from `selection_policy.scopes.explicit_pool_claim.allowed_values`; `TaskTempMailboxApplyRequest.properties.provider_name.enum` and `MailboxSessionStartRequest.properties.provider_name.enum` must be derived from `selection_policy.scopes.task_temp_apply.allowed_values`. These provider fields are nullable in OpenAPI 3.1, and their descriptions must point callers back to the owning `selection_policy.scopes.*` entry. `MailboxSessionStartRequest` must require only `caller_id` and `task_id`, must be closed to unknown properties, and must expose `source_strategy` enum values `pool_first`, `task_temp_first`, `pool_only`, and `task_temp_only`. `MailboxSessionReadRequest` must require `session_type`, `read_action`, `caller_id`, and `task_id`, must be closed to unknown properties, must expose `session_type` enum values `pool_claim` and `task_temp_mailbox`, must expose the runtime read-action enum, and must document read filters and lifecycle handles without marking both handle types as globally required. `MailboxSessionReadData` must document `session_type`, `read_action`, `email`, `status`, and `result` without documenting secret-bearing fields. `MailboxSessionCloseRequest` must require `session_type`, `caller_id`, and `task_id`, must be closed to unknown properties, must expose `session_type` enum values `pool_claim` and `task_temp_mailbox`, and must document shared length limits for `caller_id`, `task_id`, `result`, `detail`, and `reason`. `MailboxSessionCloseData` must document `session_type`, `close_action`, `status=closed`, `account_id`, `pool_status`, `task_token`, and `email` without documenting secret-bearing fields. `MailboxSessionDiscovery` must expose `start_endpoint`, `read_endpoint`, `close_endpoint`, `start_fields`, `read_fields`, `close_fields`, `read_action_values`, `source_strategy_values`, and `read_contract`.

`GET /api/external/openapi.json` must keep pool lifecycle request schemas split by action. `/api/external/pool/claim-release` uses `PoolReleaseRequest` and must not expose a `result` request property. `/api/external/pool/claim-complete` uses `PoolCompleteRequest` and must require `result` with enum values from the same runtime pool result set used by `complete_claim()`. `PoolLifecycleRequest` may remain only as a compatibility alias with `oneOf` refs to the two action-specific schemas.

`GET /api/external/openapi.json` must type the external mailbox read path for generated clients. `MessagesData` must require `emails`, `count`, and `has_more`; `MessageSummary` must require the common list/latest fields including `method`; `MessageDetail` must be an explicit typed object with content fields (`to_address`, `content`, `html_content`, `raw_content`) instead of a loose `additionalProperties` extension; `/api/external/messages/{message_id}/raw` must reference `RawMessageData`; `VerificationResult`, `AccountStatusData`, and `ProbeStatusData` must expose typed required fields matching the current controller/service payloads. These schemas may allow additional diagnostic fields where runtime providers vary, but they must not document or expose account passwords, refresh tokens, task tokens, provider JWTs, API keys, bearer token values, or provider secret values.

External request length limits are owned by `outlook_web.services.external_request_limits`. Pool APIs, task temp-mail apply/finish runtime validation, and OpenAPI request schemas must consume those constants instead of duplicating numeric limits. The task temp-mail apply `prefix` schema is a platform maximum; provider-specific prefix rules remain enforced by provider `get_options()` and runtime validation.

`selection_policy.version` is `1`.

`selection_policy.source_priority` must match `PROVIDER_SELECTION_SOURCE_PRIORITY`: `env`, `provider_config_file`, `settings`, `default`.

`selection_policy.config_file.priority_slot` is `provider_config_file`, while diagnostics that report the actual selected source use `config_file`.

`selection_policy.config_file` and `deployment_profile.config_file` must expose structured provider-config-file state: `enabled`, `env`, `formats`, `path`, `resolved_path`, `loaded`, `error_code`, and `error`. When the file loads successfully, `sections` lists the recognized top-level config sections that were present.

Provider config file failures are strict for runtime provider selection and non-strict for discovery/diagnostic display. Runtime paths such as temp-mail provider factory resolution, pool claim default resolution, and active-provider enforcement must raise `ProviderConfigFileError` for a configured but missing, unsupported, unparseable, or invalid-shaped file. Discovery and settings display paths such as `GET /api/settings`, `GET /api/providers`, `GET /api/external/providers`, `GET /api/external/capabilities`, `GET /api/external/openapi.json`, unified mailbox directory `provider_context`, and provider health/catalog lookup must remain successful and expose provider config-file error state through `provider_config_file`, `selection_policy.config_file`, or diagnostics using `source=config_file_error` plus `config_error_code` / `config_error`.

Environment overrides still take priority over provider config file errors. If a specific provider override env key is set, that env value is used and a bad `OUTLOOK_EMAIL_PROVIDER_CONFIG_FILE` must not replace it for that scope.

`selection_policy.scopes.active_allowlist` owns `ACTIVE_MAILBOX_PROVIDERS`, `active_mailbox_providers`, `active_providers`, and the empty value means all providers are active.

`selection_policy.scopes.temp_runtime_default` owns `TEMP_MAIL_PROVIDER`, `temp_mail_provider`, `runtime_temp_mail_provider`, and request override field `provider_name`.

`selection_policy.scopes.pool_claim_default` owns `EXTERNAL_POOL_DEFAULT_PROVIDER`, `pool_default_provider`, `pool_claim_provider`, `external_pool_default_provider`, and request field `provider`.

`selection_policy.scopes.task_temp_apply` points to `/api/external/temp-emails/apply` with request field `provider_name`.

`selection_policy.scopes.explicit_pool_claim` points to `/api/external/pool/claim-random` with request field `provider`.

`selection_policy.templates` must equal `deployment_profile.templates` from the same payload.

`provider_integration_guide.source_priority` must equal `selection_policy.source_priority`. Guide provider examples must be derived from catalog item `deployment`, `configuration`, diagnostics, and endpoint map rather than a separate hardcoded provider registry. GPTMail compatibility aliases and the account-level `imap` allowlist/pool alias must remain visible through the guide's alias map.

`contract_validation.version` is `1`. The validation owner is `outlook_web.services.temp_mail_provider_contract`; provider factory, provider catalog, plugin controllers, external discovery endpoints, and frontend code must consume that result instead of reimplementing provider-extension checks. `temp_mail_provider_factory.get_available_providers()` must attach static, secret-free validation to built-in and plugin temp-mail provider metadata without network probes or mailbox mutation. The authenticated plugin endpoint `GET /api/plugins/<name>/contract` may run a single-provider `get_options()` shape probe through `probe_options=true`, but it must never call mailbox create/delete/list/detail/clear methods and must not return raw options, credentials, upstream tokens, bearer values, API keys, passwords, JWTs, task tokens, or consumer keys. Validation must check that provider classes inherit `TempMailProviderBase`, then check provider key/metadata, normalized capabilities, config schema field shape, secret defaults, required provider methods, and compact `get_options()` return shape when explicitly probed. A class that only defines matching method names but does not inherit `TempMailProviderBase` must return `status=invalid`, `valid=false`, and issue code `PROVIDER_BASE_CLASS_INVALID`.

### 4. Validation & Error Matrix

- Unknown temp runtime provider -> existing temp-mail provider validation error.
- Unknown pool default provider -> settings API rejects it, or diagnostics mark it invalid.
- Inactive provider under `ACTIVE_MAILBOX_PROVIDERS` -> discovery remains machine-readable and runtime calls reject or skip according to existing active-filter behavior.
- Missing provider credential -> diagnostics report `needs_config`; discovery endpoints must not call upstream providers.
- Provider preflight without `probe_network=true` -> HTTP 200 with local readiness rows and no temp-mail provider instantiation or upstream requests.
- Provider preflight with `probe_network=true` -> HTTP 200 and only locally ready temp providers receive explicit provider-health probes; account providers remain `probe.status=not_requested`.
- Health readiness capabilities build failure -> health remains HTTP 200 if the health controller itself is reachable, `readiness.status=degraded`, `readiness.discovery.status=unavailable`, and `warnings` includes `capabilities_unavailable`.
- Mailbox directory readiness build failure -> health remains HTTP 200 if the health controller itself is reachable, `readiness.status=degraded`, `readiness.mailbox_directory.status=degraded`, and `warnings` includes `mailbox_directory_unavailable`.
- External pool disabled -> health readiness keeps discovery/provider status independent and reports `readiness.pool.status=disabled` with `external_pool_disabled` in warnings.
- Multi-key consumer without `pool_access` -> health readiness keeps discovery/provider status independent and reports `readiness.pool.status=restricted` with `pool_access_required` in restrictions and warnings.
- Multi-key consumer with `allowed_emails` -> health readiness computes `readiness.mailbox_directory` account counts after the account email scope while user-visible temp mailboxes remain counted.
- Documentation pointer update -> all discovery payloads and OpenAPI schemas must use the same documentation contract and remain HTTP 200 without checking whether each local file is present at runtime.
- Secret provider env/settings value present in runtime environment or settings -> discovery payloads may expose the key name but must emit an empty manifest hint value and must not echo the real secret.
- Temp-mail provider `config_schema` includes a secret-like field with a default value -> provider contract validation returns `CONFIG_FIELD_SECRET_DEFAULT`, provider catalog and manifest payloads remain HTTP 200, and the default secret value is omitted from safe metadata and discovery output.
- Plugin provider metadata, config schema, capabilities, or required methods are incomplete -> provider contract validation returns structured issues, discovery stays machine-readable, and plugin API callers can inspect the issue through `GET /api/plugins/<name>/contract`.
- Plugin provider class does not inherit `TempMailProviderBase` -> provider contract validation returns `PROVIDER_BASE_CLASS_INVALID`, `status=invalid`, and `valid=false`; discovery remains machine-readable so operators can inspect the invalid contract before enabling routing.
- Missing, unsupported, unparseable, or invalid-shaped provider config file -> runtime provider selection raises a structured config-file error, while discovery endpoints return HTTP 200 with `selection_policy.config_file.error_code` and diagnostics using `source=config_file_error`.
- External POST request body is valid JSON but not an object -> `INVALID_PARAM`, HTTP 400, before service-layer mutation.
- Session start `source_strategy` is unknown -> `INVALID_PARAM`, HTTP 400, before pool or temp-mail state changes.
- Session start multi-key consumer without `pool_access` uses `pool_first`, `task_temp_first`, or `pool_only` -> `FORBIDDEN`, HTTP 403, with `reason=pool_access_required`, before pool or temp-mail state changes.
- Session start `pool_first` sees external pool disabled -> fall back to task temp-mail creation through the same request body.
- Session start `pool_first` sees pool claim `NO_AVAILABLE_ACCOUNT` -> fall back to task temp-mail creation through the same request body.
- Session start `pool_first` sees public-mode `pool_claim_random` disabled -> `FEATURE_DISABLED`, HTTP 403, without fallback.
- Session start pool validation, authorization, provider configuration, upstream, or internal error -> return the original structured error without falling back to task temp-mail.
- Session start `pool_only` with external pool disabled or no available account -> return the original pool error without falling back to task temp-mail.
- Session close `session_type` is missing or unknown -> `INVALID_PARAM`, HTTP 400, before pool or temp-mail state changes.
- Session close `session_type=pool_claim` with multi-key consumer lacking `pool_access` -> `FORBIDDEN`, HTTP 403, with `reason=pool_access_required`, before pool state changes.
- Session close `session_type=pool_claim` while external pool is disabled -> `FEATURE_DISABLED`, HTTP 403, before pool state changes.
- Session close `session_type=pool_claim` and `result=release` while public-mode `pool_claim_release` is disabled -> `FEATURE_DISABLED`, HTTP 403, with `feature=pool_claim_release`, before pool state changes.
- Session close `session_type=pool_claim` and completion result while public-mode `pool_claim_complete` is disabled -> `FEATURE_DISABLED`, HTTP 403, with `feature=pool_claim_complete`, before pool state changes.
- Session close `session_type=pool_claim` with missing or non-integer `account_id` -> `ACCOUNT_ID_MISSING` or `ACCOUNT_ID_INVALID`, HTTP 400.
- Session close `session_type=pool_claim` with invalid completion `result` -> existing pool service `INVALID_RESULT`, HTTP 400, and the claim remains claimed.
- Session close `session_type=task_temp_mailbox` with unknown `task_token` -> `TASK_TOKEN_INVALID`, HTTP 404.
- Session close `session_type=task_temp_mailbox` with a task token owned by another API consumer -> `FORBIDDEN`, HTTP 403, before task mailbox state changes.
- External request field exceeds a documented length limit -> runtime rejects it before changing mailbox or pool state.
- Task temp-mail finish `result` or `detail` exceeds the documented audit-field length -> `INVALID_PARAM`, HTTP 400, and the task mailbox remains active.

### 5. Good/Base/Bad Cases

- Good: a provider is added once to the catalog/config contract and appears in `deployment_profile.provider_values` and the relevant `selection_policy.scopes.*.allowed_values`.
- Good: OpenAPI provider override enums are generated from `selection_policy.scopes.*.allowed_values`, so DuckMail or a future provider appears in clients without editing a separate schema list.
- Good: an external client reads top-level `quickstart`, sees it equals `integration_manifest.quickstart`, then uses the full manifest only for advanced workflows.
- Good: an external client reads `integration_manifest.auth`, `integration_manifest.discovery`, and `integration_manifest.providers[*].env` to generate a secret-safe `.env` scaffold without scraping the Settings UI.
- Good: an external client reads `documentation.entries.provider_onboarding.path` or `documentation.entries.openapi.endpoint` to find the human guide or OpenAPI document instead of scraping README prose.
- Good: an external integrator reads `documentation.entries.external_integration_quickstart.path`, runs `scripts/external_api_smoke.py`, and verifies discovery/OpenAPI contracts before starting mailbox sessions.
- Good: a copyable external client first reads `capabilities.data.endpoints`, falls back to canonical `/api/v1/external/*` paths only when discovery is unavailable or partial, and keeps lifecycle demos explicit about starting and closing mailbox sessions.
- Good: an external client calls `/api/external/health`, sees `readiness.discovery.next_endpoints.openapi`, `readiness.providers.summary.ready`, and `readiness.pool.status`, then decides whether to call capabilities, providers, or pool claim next.
- Good: an external client calls `POST /api/external/mailbox-sessions/start`, receives `MailboxSessionData`, then uses `next_actions.read_verification_code` and the returned lifecycle handle without choosing pool claim versus task temp-mail apply by hand.
- Good: an external client closes either returned `session_type` through `POST /api/external/mailbox-sessions/close`, so worker code does not branch into pool complete/release versus task temp-mail finish endpoints for the common path.
- Good: an external client reads `integration_manifest.workflows[key=claim_pool_mailbox].steps` to claim a mailbox, call `/api/external/messages` or `/api/external/verification-code`, then call `/api/external/pool/claim-complete` or `/api/external/pool/claim-release` without scraping README prose.
- Good: a future provider starts from `scaffold-provider`, keeps `TempMailProviderBase` inheritance, and reaches `contract_validation.status=valid` before it is enabled through `ACTIVE_MAILBOX_PROVIDERS`, `TEMP_MAIL_PROVIDER`, or `EXTERNAL_POOL_DEFAULT_PROVIDER`.
- Base: no env or config-file override is present, so settings/defaults are used and `source_priority` still documents higher-precedence sources.
- Base: an API key can use discovery and provider-backed task temp mail while pool is restricted; health reports `readiness.pool.status=restricted` without degrading provider discovery.
- Base: a restricted multi-key API consumer calls session start with `source_strategy=task_temp_only` and can create a task temp-mail mailbox without receiving pool lifecycle handles.
- Bad: adding a request field only to README or one controller payload while omitting `selection_policy`; external consumers then have to reverse-engineer behavior.
- Bad: rebuilding `quickstart` in controllers or frontend code instead of copying `integration_manifest.quickstart`; the compact starter contract can then drift from the manifest.
- Bad: silently falling back from pool authorization, provider configuration, public-mode feature-disabled, upstream, or internal errors to task temp-mail; callers lose the real reason the preferred source failed.
- Bad: implementing session close by directly updating `accounts` or `temp_emails`; this bypasses pool/task ownership checks, public-mode feature switches, audit behavior, probe cancellation, and service-level status transitions.
- Bad: returning the full `provider_diagnostics.providers` list from health; generated clients should use `/api/external/providers` for detailed provider discovery.
- Bad: adding `documentation` in one controller response but forgetting `provider_integration_guide`, `integration_manifest`, mailbox `provider_context`, or OpenAPI schemas.
- Bad: hardcoding DuckMail, Mail.tm, GPTMail, TempMail.lol, or Emailnator branches in the workflow builder instead of deriving request fields and allowed values from `selection_policy`.
- Bad: shipping an example client that hardcodes legacy `/api/external/*` paths, embeds provider bearer tokens, or starts mailbox sessions without closing them on read failures.
- Bad: putting `os.environ["DUCKMAIL_BEARER_TOKEN"]` or a stored provider API key into `integration_manifest.providers[*].env[*].value`; key names are allowed, secret values are not.
- Bad: sharing one loose `PoolLifecycleRequest` between release and complete; generated clients would think release accepts `result` or complete can omit it.
- Bad: accepting a plain Python class as a production provider just because it implements methods named `get_options`, `create_mailbox`, and `list_messages`; this bypasses the shared base-class contract for capabilities, health checks, and future extension points.

### 7. Wrong vs Correct

#### Wrong

```python
"selection_policy": {
    "source_priority": ["env", "provider_config_file", "settings", "default"],
    "templates": get_mailbox_provider_deployment_profile()["templates"],
}
```

This duplicates the priority literal and can build templates from a different deployment profile than the payload returns.

#### Correct

```python
deployment_profile = get_mailbox_provider_deployment_profile()
payload["deployment_profile"] = deployment_profile
payload["selection_policy"] = get_mailbox_provider_selection_policy(deployment_profile=deployment_profile)
```

Keep the selection policy generated from the same deployment profile and use `PROVIDER_SELECTION_SOURCE_PRIORITY` as the single source for priority ordering.

#### Wrong

```python
"PoolLifecycleRequest": {"type": "object", "additionalProperties": True}
```

This hides the difference between release and complete, and it prevents generated clients from knowing which provider fields and result values are valid.

#### Correct

```python
"PoolReleaseRequest": {"required": ["account_id", "claim_token", "caller_id", "task_id"]}
"PoolCompleteRequest": {"required": ["account_id", "claim_token", "caller_id", "task_id", "result"]}
"PoolClaimRequest": {"properties": {"provider": provider_enum_from_selection_policy}}
```

Keep each external request schema action-specific, and derive provider choices from `selection_policy` rather than hand-maintaining a second enum.

#### Wrong

```python
manifest_provider["env"] = [{"key": key, "value": os.environ.get(key, "")}]
```

This leaks runtime secret values into discovery payloads when a provider credential is configured through env.

#### Correct

```python
manifest_provider["env"] = [{"key": key, "secret": key in secret_env, "value": ""}]
```

Keep secret env/settings values empty in the manifest and expose only non-secret defaults from the catalog configuration contract.

#### Wrong

```python
data["readiness"] = {
    "providers": get_mailbox_provider_diagnostics(include_inactive=True)["providers"],
}
```

This turns health into a second provider discovery endpoint and risks leaking detailed provider state in a response that should stay compact.

#### Correct

```python
data["readiness"] = get_external_api_readiness_summary(
    consumer=get_external_api_consumer() or {},
    database_ok=db_ok,
    upstream_probe_ok=data["upstream_probe_ok"],
)
```

Keep health readiness as a secret-free summary projected from the same capabilities/provider helpers used by the dedicated discovery endpoints.

The response exposes `success`, `mailboxes`, `summary`, `facets`, `pagination`, `filters`, `provider_context`, and `contract`.

The mailbox directory contract is owned by `outlook_web.services.mailbox_directory_contract.get_mailbox_catalog_contract()`. Directory service, external capabilities, OpenAPI schemas, and frontend unified mailbox controls must consume that shared contract for `filters.kind`, `filters.status`, `filters.read_capability`, `filters.action`, `filters.sort`, `kinds`, `kind_definitions`, `status_definitions`, `read_capability_definitions`, `action_definitions`, `sort_definitions`, `summary_fields`, and `quick_view_presets` instead of maintaining separate mailbox kind, status, read-capability, action-capability, sort, summary, or recommended quick-view literals.

Mailbox source loading is owned by `outlook_web.services.mailbox_catalog.MAILBOX_SOURCE_LOADERS`. `list_unified_mailboxes()` must load records through that registry instead of directly concatenating account and temp loaders in the query flow. Each loader kind must match the ordered `contract.kinds` list, so adding a future mailbox kind has one directory loading extension point.

The authenticated unified mailbox UI must render `provider_context` as a read-only source-policy status band near the directory summary. It should show the runtime temp-mail default, pool claim default, active provider mode, local readiness counts, source priority, and the provider discovery endpoint without exposing any secret values.

`contract.version` is `1`.

`contract.item_id_format` is `{kind}:{source_id}`.

`contract.kind_definitions` describes each supported mailbox kind with `kind`, `label`, `label_en`, and `summary_key`. `contract.status_definitions` describes each supported status with `status`, `label`, and `label_en`. `contract.read_capability_definitions` describes each supported read capability with `read_capability`, `label`, and `label_en`. `contract.action_definitions` describes each supported action-capability filter with `action`, `label`, and `label_en`. `contract.sort_definitions` describes each supported sort with `sort`, `label`, and `label_en`. `contract.summary_fields` describes the summary cards and is the source for frontend summary rendering. `contract.quick_view_presets` describes recommended aggregate views with stable `key`, `label`, `label_en`, `description`, `description_en`, and a `filters` object using only `kind`, `status`, `read_capability`, `action`, `provider`, `sort`, and `search`.

`contract.quick_view_presets` must be provider-agnostic. Presets must not branch on provider keys, must keep `filters.provider` as `all` unless a future contract explicitly adds a provider-neutral provider category, and must not expose provider secret values, API keys, bearer tokens, passwords, JWTs, task tokens, refresh tokens, or consumer keys.

The authenticated unified mailbox UI may keep only minimal startup placeholders before `/api/mailboxes` returns. It must render status, read-capability, action-capability, and sort option lists from `contract.status_definitions`, `contract.read_capability_definitions`, `contract.action_definitions`, and `contract.sort_definitions`; do not keep a second complete status, read-capability, action-capability, or sort enum in JavaScript or the template.

`provider_context.version` is `1`.

`provider_context.defaults` exposes `temp_mail_provider`, `temp_mail_provider_env`, `pool_claim_provider`, `pool_claim_provider_env`, `active_mailbox_providers`, and `active_mailbox_provider_env`.

`provider_context.provider_filter` must equal the active mailbox provider filter contract, including source, mode, unknown entries, supported providers, and recognized aliases.

`provider_context.provider_diagnostics` exposes only the diagnostics summary, filter, defaults, and scope. It must not include the full per-provider diagnostics list in the mailbox directory payload.

`provider_context.readiness_summary.version` is `1`. It is the compact inventory-aware provider readiness projection for mailbox directories. It must be generated by `provider_catalog.get_mailbox_provider_readiness_summary()` from the same provider diagnostics, provider integration guide, selection policy, discovery endpoint contract, and current directory-result mailbox inventory used by `list_unified_mailboxes()`. Controllers and UI code must not rebuild this projection.

`readiness_summary` must include `overall_status`, `totals`, `issues`, `source_priority`, `provider_selector_fields`, `routing_matrix`, `capability_matrix`, `endpoints`, and `providers`. `totals` must include mailbox inventory counts (`mailboxes`, `account_mailboxes`, `temp_mailboxes`) plus provider readiness counts. The inventory counts must be computed after account-email scoping and after the current directory filters (`kind`, `status`, `read_capability`, `action`, `provider`, and `search`) are applied, so `readiness_summary.totals.mailboxes` matches the current `summary.total` / `pagination.total_count` result context rather than the full database. `provider_selector_fields.pool_claim` must come from `selection_policy.scopes.explicit_pool_claim.request_field`; `provider_selector_fields.task_temp_apply` must come from `selection_policy.scopes.task_temp_apply.request_field`. `endpoints.mailboxes`, `endpoints.providers`, and `endpoints.provider_health` must point to the external mailbox directory, provider discovery, and provider health endpoint.

`readiness_summary.routing_matrix.version` is `1`. `routing_matrix.source_priority` must match the provider selection policy source priority. `routing_matrix.scopes` must include `temp_runtime_default`, `task_temp_apply`, `pool_claim_default`, and `explicit_pool_claim`. Each scope must expose `scope`, `label`, `request_field`, `settings_key`, `env`, `endpoint`, `allowed_values`, `counts`, and `providers`. Scope provider rows must be secret-free and include `provider`, `canonical_provider`, `label`, `kind`, `active`, `configured`, `usable`, `status`, `reason`, `aliases`, and endpoint hints. Alias rows such as `gptmail` and `imap` must be resolved through the provider alias contract and integration guide instead of frontend/controller name branches.

`readiness_summary.capability_matrix.version` is `1`. The matrix is the provider-neutral capability projection for generated clients and external workers, and must be derived from provider readiness rows, provider integration guide, and routing matrix data instead of a second provider registry. It must include `generated_from`, non-negative `totals`, `workflows`, and `providers`. Workflow keys must include `mailbox_session`, `pool_claim`, `task_temp_mailbox`, `mailbox_directory`, and `provider_health`, each with `workflow`, `label`, `provider_count`, `providers`, and selector fields. Provider rows must include provider identity, kind, active/configured/readiness state, `config_source`, aliases, capability booleans, read actions, lifecycle actions, workflow support, selection fields, configuration counts, inventory counts, and endpoint hints. Capability booleans describe workflow support even when `configured=false`; configuration readiness remains represented by `configured`, `readiness_status`, and `configuration.needs_config`. The matrix must not expose provider secret values or required secret key lists.

Each `readiness_summary.providers[*]` row must be compact and secret-free: `kind`, `provider`, `label`, `active`, `configured`, `readiness_status`, `mailbox_count`, `account_count`, `temp_count`, `can_dynamic_create`, `requires_pool_inventory`, `read_capability`, `read_capabilities`, `missing_config_count`, and endpoint hints. The row source is `provider_integration_guide.providers` joined with directory inventory by `(kind, provider)`, plus inventory-only rows when a mailbox provider exists before a guide entry is available. Do not expose `provider_diagnostics.providers` inside `provider_context` to satisfy this field.

`provider_context.deployment_env`, `provider_context.deployment_profile`, and `provider_context.selection_policy` must be generated from the same provider-catalog helpers used by `/api/external/providers` and `/api/external/capabilities`, so templates and source priority cannot drift between discovery endpoints.

Temp-mail provider operation capabilities are owned by each provider class through `provider_capabilities`, normalized by `outlook_web.services.temp_mail_provider_base.normalize_provider_capabilities()`, and surfaced through `temp_mail_provider_factory.get_available_providers()`. Provider catalog code must consume that factory metadata instead of maintaining a second provider-name capability table. Stored mailbox `meta.provider_capabilities` remains the per-mailbox historical operation contract for existing records.

`provider_context.discovery.providers_endpoint` is `/api/external/providers`, and `provider_context.discovery.provider_health_endpoint` is `/api/external/providers/{kind}/{provider}/health`.

Each item must include `id`, `source_id`, `kind`, `email`, `domain`, `status`, `pool_status`, `source`, `provider`, `provider_label`, `account_type`, `read_capability`, `group`, `labels`, `remark`, `latest`, `timestamps`, `notification_enabled`, `actions`, `action_contract`, and `meta`.

`action_contract.version` is `1`. `action_contract.external` must be generated from `provider_catalog.get_external_mailbox_read_contract(lifecycle="none")`, with the item email pre-filled under each external read action's `query.email` when the source action supports an `email` query field. Fixed query values such as `mode=async` must be preserved. Do not duplicate external endpoint literals in UI code.

`action_contract.internal.open_mailbox` is the UI routing target for the authenticated app. Account-backed records use `mode=standard`; temp-mail records use `mode=temp-emails`. Internal actions may point to existing web APIs such as `/api/emails/{email}` and `/api/temp-emails/{email}/messages`, but they must remain secret-free.

Account-backed mailboxes use `kind=account`; temp-mail records use `kind=temp`.

The endpoint must not expose account passwords, IMAP passwords, refresh tokens, temp-mail task tokens, consumer keys, upstream JWTs, provider secrets, or bearer tokens.

`provider_context` must include `provider_integration_guide` and remain secret-free. Deployment templates, diagnostics, and the integration guide may include secret key names such as `DUCKMAIL_BEARER_TOKEN`, but never secret values.

Message reads remain on existing read APIs. This endpoint is an inventory/workspace contract, not a message-content contract. Directory items expose machine-readable next actions through `action_contract`; they must not embed message content.

`provider=all` returns every provider in the current context. Any other provider value is normalized to a provider key and applied as an exact provider filter. Unknown provider keys return an empty successful result so plugin-backed providers can be filtered without hardcoding a central allowlist.

`status`, `read_capability`, `action`, and `sort` are fixed enums, not arbitrary field names. Supported statuses are the `all` filter plus the values in `contract.status_definitions`. Supported read capabilities are the `all` filter plus the values in `contract.read_capability_definitions`: `graph`, `imap`, and `temp_provider`. Supported action capabilities are the `all` filter plus the values in `contract.action_definitions`: `read_messages`, `refresh_auth`, `delete_remote_mailbox`, `delete_message`, and `clear_messages`. Supported sort values are the values in `contract.sort_definitions`: `updated_desc`, `created_desc`, `email_asc`, `provider_asc`, and `status_asc`. Sorting is applied after kind/status/read-capability/action/search/provider filtering and before pagination. The response must echo the normalized values under `filters.read_capability`, `filters.action`, and `filters.sort`.

`facets.providers` lists provider filter options for the current `kind`, `status`, `read_capability`, `action`, and `search` context before the exact provider filter is applied. Each facet includes `provider`, `label`, `kind`, and `count`.

Controllers may pass generated OpenAPI metadata into the helper so `path_count`, `schema_count`, and `operation_count` match the current OpenAPI document without creating service recursion.

The canonical route is `/api/v1/external/integration-bundle`; `/api/external/integration-bundle` is a legacy alias mounted through the shared external route helper. Both routes use the same API-key auth, external API guards, response envelope, and audit behavior as capabilities, docs, health, and OpenAPI. OpenAPI `paths` must include only the canonical v1 path; legacy compatibility belongs in `legacy_endpoints`, `compatibility.aliases`, and `x-legacy-endpoints` metadata.

The bundle response `data` must include `version`, `service`, `app_version`, `status`, `generated_at`, `auth`, `endpoints`, `legacy_endpoints`, `compatibility`, `documentation`, `quickstart`, `readiness`, `provider_selection`, `openapi`, `workflows`, `smoke_checks`, `recommendations`, and `action_plan`. `auth.header` is `X-API-Key` and `auth.placeholder` is `<your-api-key>` only. `readiness.external_api` comes from `get_external_api_readiness_summary()`, `readiness.providers` comes from `get_mailbox_provider_readiness_summary()`, and `provider_selection.routing_matrix` is copied from provider readiness.

`action_plan.version` is `1`. It is the machine-readable triage layer for external workers and generated clients. It must include `status`, `summary`, and ordered `items`. Summary counters must include `total`, `blocking`, `high`, `medium`, and `low`, computed from the final deduped item list. Each item must include `key`, `priority`, `status`, `blocking`, `title`, `detail`, and at least one actionable target among `endpoint`, `command`, or `docs`. Valid priorities are `high`, `medium`, and `low`; valid statuses are `ready`, `action_required`, `optional`, and `blocked`. Remediation items such as `configure_providers`, `use_task_temp_only_or_pool_key`, `enable_external_pool`, and `probe_mailbox_directory` must appear before mutation-oriented next steps when the bundle status is `needs_config` or `degraded`. Every bundle must include `run_smoke_check` and `generate_client`; ready bundles must include `start_mailbox_session` as a ready non-blocking next step. Commands must use placeholders such as `OUTLOOK_EMAIL_PLUS_API_KEY=<your-api-key>` and `<your-base-url>` only.

Capabilities must expose a top-level `integration_bundle` discovery object plus `endpoints.integration_bundle`, `legacy_endpoints.integration_bundle`, `compatibility.aliases.integration_bundle`, `quickstart.endpoints.integration_bundle`, and `integration_manifest.discovery.endpoints.integration_bundle`. First-party docs must link and summarize the bundle. Starter clients should prefer the live bundle endpoint and fall back to local capabilities/providers/OpenAPI assembly only when the live endpoint is missing or unsupported, such as HTTP 404, 405, or 501. Starter clients must also expose a concise action-plan summary projection for external operator logs and CI gates, including bundle/action-plan status, total/blocking/priority counters, blocking keys, action-required keys, ready next-step keys, and ordered item summaries with only secret-safe endpoints, docs, or placeholder commands.

The bundle is read-only and local-only. It must not call upstream provider networks, create mailbox sessions, claim pool inventory, read messages, or emit API keys, provider bearer token values, passwords, JWTs, refresh tokens, consumer keys, mailbox credentials, claim tokens, task tokens, or provider secret values. Secret key names may appear only where the existing provider discovery contracts already allow key names.

`docs/project-launchpad.md` is the human two-minute map for GitHub visitors,
operators, and external integration developers. It must stay above the detailed
README/API/provider docs rather than duplicating those contracts in full. It must
summarize the unified mailbox workspace, supported mailbox source families,
shortest run path, local readiness gate, live external smoke path, canonical
external discovery endpoints, starter clients, and provider/plugin extension path.
It must link to `docs/external-integration-quickstart.md` and
`docs/provider-onboarding.md` for deep instructions.

The project launchpad is a required local readiness asset. Both READMEs must link
to `./docs/project-launchpad.md` near their first-start blocks, and
`scripts/project_readiness_check.py` must fail if the launchpad is missing or no
longer references the integration bundle, capabilities, providers, mailboxes,
docs, OpenAPI, mailbox session start/read/close endpoints, Python/JavaScript
starter clients, provider validation command, provider selection env keys, and
supported mailbox source families. The launchpad must be included in the same
secret scan set as README, provider onboarding, quickstart docs, config templates,
starter clients, and smoke scripts.

### 4. Validation & Error Matrix

- Missing API key -> same auth failure behavior as other external API endpoints.
- `/api/external/integration-bundle` caller -> same successful envelope semantics as `/api/v1/external/integration-bundle`.
- OpenAPI duplicates `/api/external/integration-bundle` under `paths` -> contract failure; legacy aliases belong in metadata only.
- Capabilities omits `integration_bundle` or any endpoint/alias entry -> contract failure; starter clients and smoke checks cannot discover the bundle consistently.
- OpenAPI metadata is unavailable -> bundle still returns with zero counts rather than probing upstreams or failing discovery.
- Live bundle endpoint returns 404, 405, or 501 to a starter client -> client may assemble the older local bundle from capabilities, providers, and OpenAPI.
- Live bundle endpoint returns auth, rate-limit, validation, or 5xx errors -> starter client must surface the error instead of silently falling back.
- Bundle, action plan, docs, smoke payload, or starter output contains token-like secret values -> contract failure.
- `docs/project-launchpad.md` is missing, no longer linked from either README, or
  omits key provider/API/readiness references -> local readiness failure.
- Project launchpad contains a real API key, bearer token, password, JWT, refresh
  token, task token, consumer key, provider secret, or credential-looking value ->
  local readiness secret-scan failure.

### 5. Good/Base/Bad Cases

- Good: an external worker calls `/api/v1/external/integration-bundle`, resolves blocking `data.action_plan.items`, runs `run_smoke_check`, then starts mailbox sessions only when the plan marks that step ready or intentionally non-blocking.
- Good: a GitHub visitor opens `docs/project-launchpad.md`, sees Outlook/Graph,
  IMAP, mailbox pool, built-in temp-mail providers, Cloudflare Worker temp mail,
  legacy bridge, plugin providers, canonical external endpoints, readiness
  commands, and links to the detailed quickstart/provider docs in one short page.
- Base: an older deployment lacks the live bundle endpoint, so starter clients call capabilities, providers, and OpenAPI and locally assemble a compatible planning bundle.
- Base: provider readiness is `needs_config`, so the bundle still succeeds and recommends provider configuration without exposing credential values.
- Bad: controller code manually assembles provider selection rules or readiness counters instead of calling provider-catalog helpers.
- Bad: smoke scripts validate capabilities and OpenAPI but forget to fetch the bundle, allowing endpoint-map drift.
- Bad: starter clients always rebuild a local bundle and never use the live endpoint.
- Bad: README links only to long API docs and leaves new users to infer the unified
  product shape, supported providers, and validation path by reading multiple
  deep documents.

## Scenario: Admin External API Contract Check

### 1. Scope / Trigger

Trigger: backend changes that add or modify the authenticated admin External API contract-check endpoint, local contract validation report, external smoke validator reuse, or Settings API Security contract-check UI payload.

### 2. Signatures

- `outlook_web.services.external_api_contract_check.get_external_api_contract_check() -> dict[str, Any]`
- `outlook_web.controllers.settings.api_external_api_contract_check() -> Response`
- `GET /api/settings/external-api/contract-check`
- `scripts.external_api_smoke.validate_contracts(...)`

### 3. Contracts

The admin contract check is a local-only, read-only validation report for operators before they hand the External API to another service. It is not a new external API endpoint and must not be mounted under `/api/v1/external/*` or `/api/external/*`.

Routes stay thin: `outlook_web.routes.settings` registers the URL only, `outlook_web.controllers.settings` applies login/auth and returns JSON, and `outlook_web.services.external_api_contract_check` owns report composition. The service may compose existing provider catalog, external API OpenAPI, integration bundle, readiness, and mailbox-directory service contracts. Controllers and frontend code must not rebuild provider selection, endpoint maps, OpenAPI metadata, integration manifests, quickstarts, or readiness summaries.

The report must run in process without requiring a plaintext external API key. It must not call upstream provider networks, create task temp mailboxes, claim pool inventory, read messages, mutate mailbox/database state, or audit as an external API consumer request. Any provider readiness used by the report must be local-only unless a future endpoint explicitly documents an operator-triggered probe.

The response is `{success: true, contract_check: ...}` where `contract_check` includes `version`, `status`, `generated_at`, `local_only`, `network_probes`, `mutation_safe`, `summary`, `groups`, and bounded `next_actions`. Check rows include stable `name`, `description`, `passed`, `group`, and `severity`; safe `detail` is allowed only when it cannot contain secrets.

Secret safety is part of the contract. The report may expose endpoint paths, group labels, check names, placeholder commands such as `OUTLOOK_EMAIL_PLUS_API_KEY=<your-api-key>`, and secret key names only where existing provider discovery contracts already allow key names. It must never return plaintext API keys, provider bearer tokens, provider API-key values, passwords, JWTs, refresh tokens, mailbox credentials, task tokens, claim tokens, consumer keys, or provider secret values.

### 4. Validation & Error Matrix

- Unauthenticated caller -> same login-required behavior as other `/api/settings/*` routes; do not return contract data.
- Passing local payloads -> `status=pass`, zero failed/critical counts, `local_only=true`, `network_probes=false`, and `mutation_safe=true`.
- Smoke-contract failure -> `status=fail`, grouped check rows, bounded next actions, no raw secret values.
- Unexpected service exception -> `status=error` with a safe exception type or generic detail, not provider response bodies or credential data.
- Integration bundle/OpenAPI endpoint paths drift -> contract check must fail through reused smoke validation rather than silently accepting mixed canonical and legacy paths.
- Database has no mailbox rows -> report still succeeds or fails based on contract shape, not inventory volume, and does not expose item rows beyond validation summaries.

### 5. Good/Base/Bad Cases

- Good: service builds health/capabilities/OpenAPI/bundle/provider/mailbox payloads through provider catalog and mailbox catalog helpers, wraps them in the envelope shape expected by `validate_contracts(...)`, and maps results to UI-safe rows.
- Good: the report includes `local_safety.local_only`, `local_safety.network_probes`, `local_safety.mutation_safe`, and `local_safety.secret_patterns` checks.
- Base: importing `scripts.external_api_smoke` from the service is acceptable only while the smoke module remains import-safe and has no CLI side effects.
- Bad: controller code calling `get_mailbox_provider_diagnostics()` and manually assembling provider discovery or endpoint maps.
- Bad: using `requests.get('/api/v1/external/health')` or any HTTP client from the service to validate the local app.
- Bad: returning `external_api_key`, `api_key_masked`, provider bearer token values, task tokens, or raw mailbox rows to make the UI more detailed.

### 6. Tests Required

- Backend API tests must assert authentication, success envelope shape, required report fields, row fields, local-only posture, pass/fail/error semantics when practical, and secret safety with seeded token-like settings.
- Module-boundary tests must remain green: routes import controllers only, controllers call services, services do not import Flask/routes/legacy layers, and repositories remain SQL-only.
- Frontend contract tests must assert the admin endpoint is the only fetch used by the UI contract-check panel and that helper slices do not reference Settings credential input IDs or external API endpoints.
- Readiness/smoke tests should remain green because the service composes existing contract payloads rather than creating a second source of truth.

# Scenario: External API Browser CORS Policy

## 1. Scope / Trigger

- Trigger: changes to browser access, external API discovery/readiness, or deployment environment wiring for `/api/v1/external/*` and `/api/external/*`.

## 2. Signatures

- `config.get_external_api_cors_origin_config() -> dict[str, Any]`

- `cors_config.get_external_api_cors_contract() -> dict[str, Any]`
- `cors_config.configure_external_api_cors(app: Flask) -> dict[str, Any]`

## 3. Contracts

- `EXTERNAL_API_CORS_ORIGINS` is a comma/newline-separated exact HTTP(S) origin allowlist.
- `EXTERNAL_API_CORS_ALLOW_CHROME_EXTENSION` defaults to true for backward compatibility.
- Discovery exposes safe policy metadata under capabilities/readiness `cors`; it never exposes rejected raw values.
- Credentials stay disabled. Allowed headers are `Content-Type`, `X-API-Key`, `X-Request-Id`, and `X-Trace-Id`; `X-Trace-Id` is exposed and preflight max age is 600 seconds.

## 4. Validation & Error Matrix

- Empty web allowlist plus extension enabled -> `extension_only`; ordinary web origins denied.
- Wildcard, credentials, path, query, fragment, malformed host, or non-HTTP(S) scheme -> ignored and counted in `invalid_origin_count`.
- Extension disabled plus valid web allowlist -> exact web origins still work.
- No browser origins enabled -> server-to-server readiness remains independent and must not degrade.

## 5. Good/Base/Bad Cases

- Good: `create_app()` delegates policy construction to `configure_external_api_cors()`.
- Base: canonical and legacy external paths receive identical CORS behavior.
- Bad: applying Flask-CORS globally, enabling credentials, or using `*` with API-key browser calls.

## 6. Tests Required

- Assert normalization, deduplication, invalid counting, and extension default/disable behavior.
- Assert real preflight headers on canonical and legacy paths, unknown-origin denial, internal API isolation, trace exposure, and max age.
- Assert API-key authentication still returns 401/403 independently of CORS.
- Assert capabilities/readiness remain secret-safe and project readiness detects docs/env drift.
